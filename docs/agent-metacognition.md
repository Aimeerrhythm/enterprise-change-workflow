# Agent 元认知与上下文毒化

ECW 设计哲学补充文档。探讨 agent 自我纠错能力的边界，以及这对框架设计意味着什么。

---

## 核心问题

Agent 能否在没有外部信号的情况下，主动意识到"我当前的某个前提可能是错的"？

结论：**不能，或非常不可靠。** 这不是工程问题，是当前架构的认识论约束。

---

## 上下文毒化的两种形态

**外部投毒**：工具返回值或用户输入中含有恶意指令，诱导 agent 执行不应执行的操作。这是安全问题，已有较多讨论。

**内部自毒**：更隐蔽。Agent 在推理过程中形成错误前提，所有后续推理都在这个前提上叠加——越推越偏，越偏越自信，因为内部逻辑是自洽的。

内部自毒是本文关注的对象。

---

## 为什么自我否定很难

### 架构原因

Transformer 的注意力机制天然倾向于"找支持"而非"找反例"。一旦错误前提出现在上下文里，后续 token 的生成概率就已经被它条件化。上下文越长，这个前提被"重复确认"的次数越多，权重越大。

这是架构特性，不是 prompt 能完全修复的。

### 雪球效应

```
错误前提 P
  → 基于 P 的推理 R1（局部自洽）
    → 基于 R1 的结论 C1
      → C1 被当作新前提，继续推理 R2
        → ...
```

每一步局部自洽，整体持续偏离。更危险的是：**越往后，支持错误前提的 token 越多，模型越"确信"**。错误积累的同时，置信度也在上升。

### 真正的元认知需要什么

元认知需要三个条件：

1. **表征自己的信念状态**——知道自己"相信"什么
2. **评估信念的可靠性**——知道这个信念来自哪、有多脆弱
3. **主动搜索反例**——不是等反例出现，而是主动去找

LLM 对第 1 条做得尚可，对第 2 条很弱，对第 3 条几乎没有。

## 补丁的根本局限

### 所有补丁运行在同一个基础设施上

| 补丁手段 | 制造的摩擦 | 失效场景 |
|---------|-----------|---------|
| 工具结果验证 | 现实 vs 预期不一致 | 错误前提影响了对"正常结果"的判断 |
| 对抗子 agent | 独立上下文的异见 | 相同模型有相同系统性盲点，可能互相"验证" |
| 置信度标注 | 不确定性显式化 | 置信度信号本身被毒化，越错越自信 |
| checkpoint 前提列表 | 前提显式记录 | Agent 不知道某前提存在，就不会把它写进去 |

**根本问题**：所有机制都依赖 agent 在某个时刻"知道自己可能错在哪"。它不知道的那部分，所有补丁都照不到。

### Unknown Unknowns 问题

```
Known unknowns：agent 知道自己不确定
  → 补丁可以处理（标 confidence: medium，触发人工审查）

Unknown unknowns：agent 不知道自己不知道
  → 补丁触达不到
```

上下文毒化最危险的形态恰好是 unknown unknowns——agent 完全没有意识到某个前提是可疑的，因为它从未进入"需要质疑"的候选集。

### 自我评估的自引用困境

用来评估前提可靠性的，是同一个被毒化的 agent。如果前提 P 是错的，agent 基于 P 生成的"confidence: high"也可能是错的。自我评估的可信度与被评估内容的可信度耦合在一起，无法用自身来校验自身。

## 设计方向的根本转变

> 不是"让 agent 更聪明地发现自己的错误"  
> 而是"假设 agent 一定会犯某类错误，系统怎么设计才能让这些错误代价最小"

这意味着：

**按可逆性划分决策，而不是按重要性。**

- **可逆的错误**：让 agent 犯，犯了改，成本低。不需要在这里堆补丁。
- **不可逆的决策**：不信任任何自动机制，强制人介入。这个边界要明确划出来，而不是期望 agent 自己识别。

堆补丁让系统看起来更可靠，但可能只是把失败模式变得更不可见。清晰的人工节点虽然"低效"，但失败时可以被发现和修正。

---

## 对 ECW 设计的含义

**现有机制中做对的部分：**

- `ecw:spec-challenge` 使用独立上下文的 agent 挑战主 agent 结论，切断了毒化传播链——方向正确，但受制于同模型盲点问题
- `phase2-assessment.md` 把前提和结论显式写下来，供后续 phase 冷启动对照——比隐式上下文传递更可靠
- PreCompact hook 在上下文压缩（高风险节点）前介入——防止反驳证据被压缩丢失

**值得进一步探索的方向：**

**前提显式化 + 主动 falsify**：不只记录结论，还记录"支撑这个结论的关键前提"和"什么样的证据会推翻它"。impl-verify 阶段主动去寻找这些反例，而不是只验证实现是否符合计划。

```yaml
premises:
  - content: "OrderService 不直接调用 InventoryService"
    source: "cross-domain-calls.md L23"
    confidence: high
    falsifiable_by: "如果发现 OrderService 有直接 import InventoryService"
  - content: "MQ 消息是幂等消费的"
    source: "推断（无文档确认）"
    confidence: medium
    falsifiable_by: "如果 consumer 代码没有去重逻辑"
```

**"假说"而非"结论"的语义契约**：科学方法里没有结论，只有"暂时未被证伪的假说"。如果 agent 的输出从语义上被定义为假说，下游使用者会自然保持质疑姿态，而不是在 agent 说完后切换到执行模式。

---

## 延伸思考

以下问题尚无定论，但值得持续关注：

**置信度信号的可信性**：LLM 的置信度表达与实际准确率之间校准很差，且在上下文毒化情况下错误越深越自信。所有依赖置信度来决定是否升级人工审查的机制，可能在最需要的时候失效。

**对抗场景下的彻底崩溃**：以上讨论都假设善意环境。在对抗性场景（恶意 prompt 注入、伪造工具返回值）中，所有依赖 agent 自我报告的机制（置信度、前提列表、自我审查）都变成攻击面而非防御面。

**人不是补丁，人是结构**：把"人介入"视为 agent 失败时的兜底，本身就是错误的框架。人类是这个系统里唯一真正具备元认知能力的节点。更准确的定位是：agent 是人的认知的延伸工具，元认知始终在人这里。系统设计目标不是让 agent 更自主，而是让人的元认知能更高效地作用在更大的范围上。
