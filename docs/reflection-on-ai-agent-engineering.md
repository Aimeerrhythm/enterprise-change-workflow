# Prompt 工程的可靠性天花板：当"代码"是一段自然语言时

## 0. 一个被忽视的前提

传统软件工程建立在三个隐含假设之上：

1. **程序的载体是形式语言**——其语义由编译器或解释器精确定义
2. **执行是确定性的**——相同输入产生相同输出
3. **正确性是可判定的**——至少在逻辑层面，一个程序"对不对"有确定答案

ECW 项目把这三点全部打破了。

它的"程序"核心是 15 个 SKILL.md 文件——用自然语言写的指令集，交给 LLM 执行。16 天内迭代了 295 个 commit，其中 58 个是 fix。这不是因为开发者能力不足，而是因为执行模型本身的结构性缺陷让"正确性"变成了一个统计概念。

这篇文章试图回答一个问题：**当一个软件系统的执行引擎是概率性的、编程语言是自然语言、编译器不存在的时候，"构建正确的功能"到底意味着什么？**

---

## 一、两类任务，一种载体——所有问题的起点

Prompt 在 ECW 中被用于执行两类性质完全不同的任务：

**认知任务（Cognitive）**：理解需求文本、分析代码影响面、生成实现方案、发现边界条件。特征是模糊输入 → 合理输出，80% 准确已有价值。

**确定性任务（Deterministic）**：路由决策、状态转移、marker 格式校验、契约验证。特征是精确输入 → 唯一正确输出，99% 正确意味着 1% 概率系统崩溃。

这两类任务的容错模型完全不同。认知任务允许"有时不准确"，不准确的代价是方案质量下降。确定性任务不允许"有时错误"——一次路由决策错误意味着整条 skill 链的执行路径全部偏离，后续所有 skill 的上下文注入、契约检查、门禁触发全部基于错误的前提运行。

但 ECW 将两类任务编码在**同一种载体**里：自然语言 prompt。它们通过同一种执行机制运行：LLM 的 next-token prediction。

```
"P0 跨域需求 → domain-collab → Phase 2 → writing-plans → spec-challenge"
```

这是一个完全确定的路由规则，没有任何模糊空间。但在 prompt 系统里，它的"执行"意味着 LLM 读取这段文字、理解其含义、在输出中复现这个决策。任何一个环节都可能漂移。

反直觉的结论：**用更聪明的模型不会解决这个问题。** 确定性逻辑需要的不是"更聪明的概率引擎"，而是"确定性引擎"。10 倍推理能力的 LLM 对 `if risk == P0 then route = [...]` 的执行，永远不如一行 Python 代码可靠。原因不在于模型不够强，而在于概率性执行和确定性逻辑之间存在不可消除的阻抗失配。

## 二、编译器缺位——不只是工具缺失，是保障机制的消失

传统软件工程里最被低估的组件不是任何框架或语言特性，是编译器。

编译器提供的不是"翻译"，是**局部修改的全局安全性验证**。修改一个函数签名 → 编译器立即枚举所有调用方的不兼容。删除一个字段 → 类型检查器在运行前给出所有访问点的精确位置。改变一个接口 → 所有实现者必须同步适配。

这套保障机制如此基础，以至于大多数开发者意识不到它的存在——就像意识不到呼吸的空气。

Prompt 系统完全没有这套机制。修改 risk-classifier 中一段关于路由的描述：

- 没有机制告诉你 domain-collab 的 prompt 中有一个隐式依赖对应这段描述的行为
- 没有机制验证 session-state.md 的写入格式是否与 auto-continue hook 的正则匹配
- 没有机制枚举"所有假设 Next 字段值为 `ecw:writing-plans` 的下游逻辑路径"

grep 能搜索文本，搜索不到语义依赖。"Phase 2 完成后立即触发下一个 skill" 和 "将 session-state.md 的 Next 字段更新为 ecw:writing-plans" 之间的因果链，文本搜索工具完全感知不到。

ECW 的 849 个 static test 填补了一小部分——验证了 marker 格式一致性、路由表完备性、文件引用完整性。但这只覆盖了**结构正确性**，覆盖不了**行为正确性**。HTML validator 能告诉你标签正确闭合，但告诉不了页面功能是否正常。

结构正确性和行为正确性之间的鸿沟，在 prompt 系统里比在传统软件里宽得多。传统软件里"结构正确"（通过编译）+ "单元测试通过"通常能带来很高的行为信心。Prompt 系统里，结构正确只是一个必要条件，离充分条件还有巨大距离。

## 三、隐式契约——分布式系统中的"总线"没有 schema

ECW 的 15 个 skill 通过 `session-state.md` 通信——这个文件是整个系统的"总线"。

总线承载了大量契约：

- `Auto-Continue` 字段必须是 `yes`/`no`（auto-continue hook 的正则依赖此格式）
- `Next` 字段必须包含 `ecw:` 前缀（否则下游 skill 无法被正确路由）
- `Routing` 字段的分隔符是 `→` 而不是 `->`（hook 的 split 逻辑按箭头符号切割）
- MODE marker 必须精确写作 `<!-- ECW:MODE:START -->` 而非 `<!-- ECW:MODE -->`（session-start.py 和 pre-compact.py 各自实现了不完全一致的解析逻辑）
- Subagent Ledger 的列顺序固定（先 Started 后 Duration），cost-tracker 的表格解析依赖列序

这些契约没有一个集中的 schema 定义。它们散落在 15 个 SKILL.md、15 个 Python hook 脚本里。每个 skill 从自身视角描述"我期望读到什么格式的 session-state"，但没有一个地方定义"session-state.md 的完整契约"。

这正是传统软件用 interface、schema、type system 精确解决的问题：**契约显式化、集中定义、机器可验证。**

session-state.md 在 15 个 skill 中被引用了 40+ 次，是事实上的系统总线，但从未被当作总线来设计。

后果是系统性的：修改一个 skill 的输出格式，无法可靠地枚举所有受影响的消费者。测试只能覆盖已知依赖，覆盖不了隐式耦合。每一次跨 skill 改动都携带不可量化的回归风险。

ECW 后来引入了 `data_contracts.yaml` 和 workflow simulator。这本质是在手工构建一个"类型系统"——为自然语言 prompt 之间的数据流提供编译期检查。但它只能回答"如果每个 skill 严格按契约执行，结构是否自洽"，回答不了"LLM 是否会按契约执行"。

## 四、概率性漂移——"正确"从布尔值变成了概率分布

传统软件中，`f(x)` 的行为是一个点：给定输入，确定输出。

Prompt 系统中，LLM 对同一段 prompt 的执行是一个概率分布。这是两者之间最深刻的认识论差异，产生三个连锁后果：

**第一，测试通过 ≠ 行为正确。** 传统测试的语义是"该用例下程序行为正确"。prompt eval 的语义是"该次采样中 LLM 输出符合预期"。下一次采样可能不符合。同一个 eval case 连跑 10 次全部通过，第 11 次可能触发不同的 token 生成路径。

**第二，Bug 不可稳定复现。** 传统 bug 的复现条件是确定性的——给定这个输入状态，bug 必然触发。prompt bug 的触发带有概率性——同样输入有时触发有时不触发。这直接动摇了调试的基础。

**第三，修复无法被验证。** 传统修复的验证路径：复现 bug → 应用修复 → 确认不再触发 → 回归测试通过。prompt 修复的验证是"跑了 N 次都没出错"，永远不能排除第 N+1 次出错的概率。

这三个后果直接冲击了开发者的确定性感受。传统开发的体验是"找到问题 → 解决问题 → 确认解决"。prompt 开发的体验是"发现了问题 → 调整了 prompt → 看起来好了 → 但心里没底"。

"改了出错、错了再改"的循环不是流程问题，是验证手段的结构性缺失。当一个系统无法提供"修复已确认"的确定性信号时，反复修改是唯一理性的行为——你永远在试图逼近那个不知道是否存在的"这一次真的对了"的状态。

## 五、"更仔细"的陷阱——加更多 prompt 指令是负期望策略

面对以上问题，直觉反应是：在 SKILL.md 里加更多约束——更多的 MUST，更多的 NEVER，更多的 CRITICAL，更多的 error handling 分支，更多的 Common Mistakes 列表。

这本质是**用更多的 prompt 修复 prompt 导致的问题**。有三个层面的反效果：

**注意力稀释。** 当 500 行 prompt 里有 15 个 CRITICAL 标记时，没有一个会被真正当作 critical。强调词堆叠产生"狼来了"效应——所有东西都重要 = 没有东西重要。

**指令冲突。** 自然语言没有编译器的约束消解能力。一条防御性指令（"遇到 X 情况要先检查 Y"）可能与另一条流程指令（"不要中断 Phase 1 的执行流"）在特定场景下产生矛盾，LLM 需要自行调和，而这个调和过程本身就是概率性的。

**复杂度正反馈。** 每加一条防御性指令 → prompt 更长 → 注意力更分散 → 执行更不稳定 → 更需要防御性指令。这是一个自我强化的恶性循环。

这个陷阱的核心在于混淆了两种不同的保障机制：**约束**（constraint）和**提示**（suggestion）。传统软件的约束是硬性的——编译不过就是过不了。Prompt 系统的"约束"本质上是措辞程度的差异——MUST 和 should 的区别在于语气而非执行保证。把硬约束的需求编码为软提示的格式，是 prompt 工程里最普遍的范畴错误。

## 六、从 prompt 到 hook——确定性机制化的逻辑

ECW 演进中最具启发性的决策，是把 skill 之间的自动过渡从 prompt 层迁移到 Hook 层。

最初，自动衔接靠嵌入在每个 SKILL.md 中的 CRITICAL 指令实现：完成当前阶段后不要问用户确认，直接 invoke 下游 skill。这段指令在 9 个文件中以不同措辞重复了 14 次，有时生效有时不生效。

v1.2.5 将此逻辑迁移到 `auto-continue.py`——一个 PostToolUse hook：

1. ECW skill 加载完成 → PostToolUse 事件触发
2. Hook 用正则解析 session-state.md 的 `Auto-Continue` 和 `Next` 字段
3. 若 Auto-Continue 为 yes 且 Next 有值 → 通过 systemMessage 注入继续指令
4. 此决策不再依赖 LLM 是否"注意到"某段 prompt 中的 CRITICAL 标注

差异不在代码量。在一个是概率性的（LLM 可能遵守也可能忽略 prompt 指令），一个是确定性的（Python 对相同输入必然产生相同分支）。

更一般的规律：**当一段行为规则在 prompt 中重复出现 3 次以上、使用 MUST/CRITICAL/NEVER 等强调词、且违反它会导致工作流损坏——三条全中时，这条规则不应该在 prompt 里。它应该是一个确定性机制。**

强调词不是约束信号，是**约束不足的补偿信号**。当一段指令需要 CRITICAL 来强调，恰恰说明它在当前形式下无法被可靠执行。

## 七、上下文毒化——已知错误和未知错误的分界线

Agent 能在多大程度上识别自身的错误前提？答案是：**几乎不能，或者说非常不可靠。**

Transformer 的注意力机制天然倾向于"寻找支持性证据"而非"寻找反例"——这是架构特性，不是 prompt 能修复的。上下文越长，错误前提被"重复确认"的次数越多，在后续 token 生成中的条件化权重越大。错误在积累，置信度也在上升——这两个过程是耦合的。

由此产生一个对框架设计至关重要的区分：

**Known unknowns**：agent 知道它不确定。补丁可以处理（标记 confidence: medium，触发人工审查）。

**Unknown unknowns**：agent 完全不知道它应该是不知道的。补丁触达不到。上下文毒化最危险的形态恰好是这一类——agent 未曾将某个前提纳入"需要质疑"的候选集。

ECW 的 spec-challenge（独立上下文 agent 做对抗性审查）在原理上切断了毒化的传播路径。但有一个无法消除的限制：**同模型有同系统性盲点**。用 Opus 审查另一个 Opus 的方案，两个实例可能同时遗漏同一类边界条件。

更根本的限制：**自我评估的可信度与被评估内容的可信度是耦合的，系统无法用自身校验自身。** 如果前提 P 是错的，基于 P 生成的 confidence: high 同样不可信。

这对框架设计的启示：**纠错机制不应和主要执行逻辑共享同一个认识论基础。** 外部信号（人工判断、确定性脚本的校验结果、独立模型提供商的 agent）比内部信号（同模型 self-review、confidence estimate）可靠得多——不是因为外部信号更聪明，而是因为它不携带相同的系统性盲点。

## 八、自然语言没有作用域——复杂度爆炸的隐藏驱动

传统编程语言有一个被严重低估的特性：**作用域**。函数内的局部变量不影响外部，模块的内部实现对外部不可见。作用域提供了**隔离**和**封装**两层关键保证。

自然语言 prompt 两者皆无。

在 500 行 SKILL.md 末尾添加一段 error handling 说明，可能通过注意力机制微妙地改变 LLM 对开头 Phase 1 步骤的执行方式。无法声明"这段文字只作用于 Phase 2"——LLM 不按人类划分的段落边界分配注意力权重。

SKILL.md 中的所有内容对 LLM 都是全可见的、可交互的。调用一个 skill 意味着 LLM 读取该 skill 的全部 prompt 内容——不存在"接口"和"实现"的分离。

prompt 系统的复杂度增长曲线比传统软件更陡——每增加一个 skill，与所有已有 skill 的语义交互面是 O(N)，而且这种交互不可见、不可枚举。传统的模块化手段（分文件、分子目录）提供了组织结构上的清晰性，但提供不了执行层面的隔离性。

## 九、"调试"的语义已经变了

传统调试的前提是**错误可归因**：stack trace → 调用链 → 根因 → 修复，每一步的因果关系确定。

Prompt 系统的调试面临根本障碍：**只能看到输出，看不到推理路径。** 无法获知 LLM 从哪一步开始偏离、把哪段指令理解错了、哪些指令在内部表征中产生了矛盾。

调试变成了猜谜游戏——大量精力不是花在修复问题上，而是花在猜测问题在哪上。猜错的代价很高——基于错误归因的修复方向会引入新问题，形成 fix → break → fix 循环。

## 十、Eval 的承诺与局限

业界主流叙事：eval 之于 prompt 工程，犹如单元测试之于传统编程。

这个类比在某个层面上成立——eval case 能捕获"修改是否破坏了已知的期望行为"。但在更深层面上会误导。

单元测试和编译器是互补的。编译器提供**完备性**（覆盖所有代码路径的结构正确性），单元测试提供**指定路径上的行为验证**。两者缺一不可。

Eval 试图同时扮演两种角色，但两种都做不好：作为编译器，它是采样的不是完备的；作为测试，结果是统计性的不是确定性的。"通过"是统计判断，不是逻辑判断。

更准确的定位：**eval 是统计采样器，不是正确性证明器。** 它给出的信息是"在 N 次采样中 M 次符合预期"，而不是"这个系统是正确的"。两者之间的差异不在于 N 不够大，而在于统计判断永远不是逻辑证明。

## 十一、放大器还是拐杖——一个评估框架

面对模型能力的持续提升，prompt 工程的组件可以按一个标准划分：**它编排的是模型做不到的事，还是补偿模型暂时做不好的事？**

**放大器（Amplifier）**：编排模型自身无法实现的能力——风险治理流程、审计追踪、多角色对抗审查、组织级合规约束。模型越强，放大器价值越大。

**拐杖（Crutch）**：补偿模型当前的能力缺口——分步思考引导、格式强制约束、关键词路由表、context window 变通方案。模型提升后直接贬值。

实践意义：在放大器上投入架构资源，拐杖用最小复杂度实现、保持可替换。每次增加复杂度时追问一个问题：**10 倍聪明的模型来了，这一层还有用吗？**

## 十二、走向生产——需要什么基础设施

将 prompt-based workflow 部署到生产环境，面临三个基础设施缺位：

**可重复性。** 传统 CI/CD：同一 commit + 同一 runner → 同一产物。Prompt 系统做不到——同一 SKILL.md + 同一输入，两次执行可能走不同决策路径。

**可审计性。** "为什么做了这个决策"——传统软件有确定代码路径 + log + trace。Prompt 系统里决策过程封装在 LLM 内部表征中，session-state.md 记录了结果，记录不了推理路径。

**可回滚性。** 传统回滚：部署上一版本代码 → 行为完全恢复。Prompt 系统回滚文本不难，但无法保证 LLM 行为完全恢复——即使文本相同，概率性意味着行为可能不同。

改善路径：把确定性逻辑（路由、状态转移、格式校验、契约验证）剥离到代码层；prompt 只负责理解、分析、生成、发现这些天然容忍概率性的任务；在 prompt 输出和确定性执行之间建立明确闸门。

## 十三、框架 > 技巧

当前 prompt 工程讨论大多聚焦"怎么写出更好的 prompt"——措辞技巧、结构模式、few-shot 策略。

这相当于在没有编译器和高级语言的年代讨论"怎么写出更少 bug 的汇编代码"。

真正的杠杆在上一层：**prompt 之上的治理框架。**

**类型系统**——data contracts，集中定义组件间的数据契约，可自动验证。

**模块化**——流程与 prompt 分离，路由逻辑与推理指令分离，不同生命周期的关注点不互相污染。

**确定性编排**——识别 prompt 系统中的确定性子任务，迁移到代码层。代码做骨架，prompt 在骨架内填充智能。

**契约验证**——结构验证防格式级崩溃，行为采样（eval）做回归检测。两者加起来，仍然不是编译器，但各自提供了当前技术条件下可达到的最强保证。

ECW 的价值不在于 15 个 SKILL.md 能跑通。在于它在尝试做一件更长半衰期的事：**在概率性引擎之上，重新构建一层确定性的工程约束。**

模型会升级，prompt 技巧会过时。但架构层面的原则——确定性优先于概率性、契约显式化、关注点分离、风险比例驱动、人的元认知作为系统的结构性组件而非兜底补丁——这些会因为解决的"用自然语言构建软件系统"的结构性问题而持续成立。
