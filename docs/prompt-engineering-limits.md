# Prompt 工程的可靠性天花板

ECW 设计哲学补充文档。探讨当 prompt 被用于构建复杂系统时遇到的结构性困境，以及这对 AI Agent 工程化的含义。

起因：ECW v1.2.x 阶段陷入"改了出错、错了再改"的循环。295 个 commit 中 58 个是 fix（20%）。这不是粗心导致的，是一个更深层问题的症状。

---

## 两类任务，一种载体

Prompt 被用于两类本质不同的任务：

| | 认知任务 | 确定性任务 |
|--|---------|-----------|
| **例子** | 理解需求、分析代码影响、生成方案 | 路由决策、状态转移、marker 格式校验 |
| **本质** | 模糊输入 → 合理输出 | 精确输入 → 唯一正确输出 |
| **容错** | 80% 正确已经很有价值 | 99% 正确意味着 1% 的时候系统崩溃 |
| **LLM 适合度** | 高——这是 LLM 的核心能力 | 低——概率性引擎执行确定性逻辑 |

ECW 当前将两类任务都编码在 SKILL.md 的自然语言里。这意味着路由决策（"P0 单域走 requirements-elicitation → Phase 2 → writing-plans → spec-challenge"）和需求分析（"理解用户想要什么"）使用同一种执行引擎——LLM 的 next-token prediction。

一个走错路由，整条链路就偏了。而路由逻辑本身是完全确定性的——给定 risk level + domain mode + change type，输出唯一的 skill chain。这种事不应该让概率性引擎来做。

---

## 编译器缺位

传统软件开发有一个隐含假设被严重低估了：**编译器保证了局部修改的全局安全性。**

你改了一个函数签名，编译器立刻告诉你所有调用方需要同步修改。你改了一个类型定义，类型检查器在你运行代码之前就列出所有不兼容的地方。你删了一个方法，IDE 的引用搜索精确到行号。

Prompt 系统没有这些。改了 risk-classifier 里的一句话：

- **没有编译器** 告诉你 domain-collab 里有一个隐式依赖这句话的行为
- **没有类型检查器** 验证 session-state.md 的字段格式是否还和 auto-continue hook 的正则匹配
- **没有引用搜索** 能精确定位"所有假设 Next 字段存在的代码路径"

有的是 grep。但 grep 只能搜文本，搜不到语义依赖。"After outputting Phase 2 report, invoke the next skill" 和 "update session-state.md Next field" 之间的因果关系，grep 看不见。

**ECW 的 static test 填补了一小部分空缺**——它验证了 marker 格式、路由表一致性、文件引用完整性。但这只覆盖了"结构正确性"，覆盖不了"行为正确性"。就像 HTML validator 能告诉你标签闭合了，但告诉不了你页面是否能用。

## 隐式契约问题

ECW 的 15 个 skill 之间通过 session-state.md 通信。这个文件承载了大量隐式契约：

- `Auto-Continue` 字段必须是 `yes`/`no`，auto-continue hook 的正则才能匹配
- `Next` 字段必须包含 `ecw:` 前缀的 skill 名
- `Routing` 字段的 `→` 分隔符不能换成 `->`
- MODE marker 必须用 `<!-- ECW:MODE:START -->` 而不是 `<!-- ECW:MODE -->` 
- Subagent Ledger 的列顺序和列数必须固定

这些契约没有 schema 定义，散落在 15 个 SKILL.md 和 15 个 hook 脚本里。每个 skill 各自描述"自己需要怎么写 session-state.md"，但没有一个地方定义"session-state.md 的完整契约是什么"。

后果：改一个 skill 的输出格式时，你不知道哪些下游消费者会受影响。这正是传统软件用 interface/schema/type 解决的问题——**让契约显式化、集中定义、机器可验证。**

session-state.md 在 15 个 skill 中被引用了 40+ 次。它事实上是整个系统的总线，但没有被当作总线来设计。

---

## 概率性漂移

即使所有契约都显式化了，prompt 系统还有一个传统软件不存在的问题：**同一段 prompt，不同次执行的行为可能不同。**

传统函数 `f(x)` 对相同输入总返回相同输出。但 LLM 对相同 prompt 的响应是概率性的。这意味着：

1. **测试通过不代表行为正确**——这次通过了，下次可能走不同分支
2. **Bug 不可稳定复现**——同样的输入，有时触发有时不触发
3. **修复不可验证**——你改了 prompt，跑了 5 次都对了，第 6 次可能又错了

这是 prompt 工程和传统编程之间最根本的认识论差异。传统软件的正确性是可证明的（至少在逻辑层面）；prompt 系统的正确性是统计性的——你只能说"在 N 次测试中 M 次正确"，永远不能说"它是对的"。

这直接影响了开发者的心理状态。传统开发者改完代码，跑完测试，有一种"改完了"的确定感。Prompt 工程师没有。**你永远不确定系统是否真的修好了，还是只是这几次碰巧没出问题。** 这就是"改了出错、错了再改"循环的心理根源——不是能力不足，是验证手段的根本性缺失。

## 复杂度的非线性爆炸

传统软件的复杂度增长大致是可管理的：N 个模块的交互面是 O(N²)，但通过分层、接口隔离、依赖注入等手段，实际耦合可以控制在 O(N) 量级。

Prompt 系统的复杂度增长更陡。原因：

**自然语言没有作用域。** 编程语言里的变量有明确的作用域边界——函数内的局部变量不会影响函数外。但 prompt 里的每一句话都可能影响 LLM 对整个 prompt 的理解。在 risk-classifier 的 SKILL.md 末尾加了一段 error handling 说明，可能微妙地改变了 LLM 对 Phase 1 步骤的执行方式——因为 LLM 的注意力机制会让所有 token 互相影响。

**没有封装。** 传统编程的 class/module/function 提供了信息隐藏。调用方不需要知道实现细节，只需要知道接口。但 SKILL.md 里的所有内容对 LLM 都是可见的、可交互的。你不能说"这段 prompt 只影响 Phase 2，和 Phase 1 无关"——LLM 没有这种隔离机制。

**调试信息损失。** 传统软件出错时有 stack trace、断点、内存 dump。Prompt 系统出错时，你看到的是 LLM 的最终输出，但看不到"它在哪一步开始偏离的"、"它把哪句 prompt 理解错了"、"哪两段 prompt 产生了矛盾的指引"。**错误是沉默的、延迟的、难以归因的。**

ECW 当前 2857 行 SKILL.md + 15 个 hook 脚本 + 20 多个辅助文件，已经处在这种非线性爆炸的区域内。

## "更仔细"的陷阱

面对这些问题，最直觉的反应是"下次更仔细"。这是一个陷阱。

"更仔细"的具体表现通常是：在 SKILL.md 里加更多的 MUST / NEVER / CRITICAL 标注，加更多的 error handling 分支，加更多的"Common Mistakes"列表。

这些做法的本质是**用更多的 prompt 来修复 prompt 导致的问题**。但：

- 更多的 prompt = 更大的注意力竞争面 = 更高的概率漂移风险
- MUST/NEVER/CRITICAL 的堆积会产生"狼来了"效应——当所有东西都是 CRITICAL，就没有东西是 CRITICAL
- 每加一条防御性指令，就多了一条可能和其他指令冲突的指令

这正是 `design-principles.md` 里"确定性优先于概率性"原则的核心洞察：**如果某件事必须可靠发生，它不应该是一条 prompt 指令，而应该是一个确定性机制。**

ECW 已经在这个方向上走了一步：auto-continue 从 9 个 SKILL.md 里的重复 prompt 块变成了一个 Python hook。但还有大量的确定性逻辑仍然编码在 prompt 里。

## 生产环境的适应性问题

把以上所有问题放到生产环境的语境下，情况更严峻：

**可重复性要求。** 生产环境要求同一个输入产生一致的输出。传统 CI/CD 能做到：同一个 commit 在任何机器上构建出相同的产物。Prompt 系统做不到——同一个 skill 对同一个需求，今天跑和明天跑可能走不同的路径。

**可审计性要求。** 生产环境出问题时需要事后追溯"为什么做了这个决策"。传统软件有 log、有 trace、有确定性的代码路径可以复现。Prompt 系统的决策过程封装在 LLM 内部，不可复现、不可追溯。session-state.md 记录了结果，但记录不了推理过程。

**可回滚性要求。** 传统软件发现问题可以回滚到上一个版本，行为完全恢复。Prompt 系统改了一个 SKILL.md，回滚文本不难，但你无法保证"回滚后 LLM 的行为和之前一模一样"——因为即使文本完全相同，模型的概率性意味着行为可能不同。更不用说模型供应方还会在后台更新模型权重。

## 这意味着什么

以上不是在否定 prompt 工程的价值，也不是在否定 ECW 的方向。它们是在划定一条边界：

**Prompt 适合做什么：**
- 理解模糊的自然语言输入
- 生成结构化的分析和建议
- 在给定框架内完成创造性任务（写代码、写方案）
- 在人工监督下做决策辅助

**Prompt 不适合做什么：**
- 作为确定性状态机的实现载体
- 保证多组件之间的契约一致性
- 提供可复现、可审计的执行路径
- 在无人监督下做高后果决策

ECW 的困境不是它做了错误的事情，而是它用错误的工具做了正确的事情中的一部分。路由编排、状态管理、契约校验——这些是"正确的事情"，但用 prompt 实现它们是"错误的工具"。

这指向一个架构层面的演进方向：**分离关注点——让代码做确定性的事，让 prompt 做认知性的事。** 两者不是替代关系，是协作关系。代码提供骨架和护栏，prompt 在骨架内填充智能。

这不是 ECW 特有的问题。任何试图用 prompt 构建复杂多步骤 workflow 的系统都会在某个复杂度阈值上遇到同样的困境。区别只在于这个阈值是 5 个 skill 还是 15 个——但边界一定存在。

认识这个边界，不是放弃，是为了更聪明地分配力量。

## 延伸思考

**Eval 是 prompt 工程的编译器吗？** 部分是。Eval case 能捕获"这次改动是否破坏了已知的期望行为"，类似回归测试。但它和编译器有根本区别：编译器是完备的（覆盖所有代码路径），eval 是采样的（只覆盖你写了 case 的路径）。而且编译器的结果是确定的（通过/不通过），eval 的结果是统计的（N 次中 M 次通过）。Eval 是必要的，但不能提供编译器级别的信心。

**模型升级会解决这些问题吗？** 部分会——更强的模型能更可靠地遵循复杂指令，缩小概率漂移的方差。但不会根本解决。原因见 `design-principles.md` 的"模型升级测试"：确定性逻辑需要的不是"更聪明的概率引擎"，而是"确定性引擎"。10 倍聪明的模型对 `if risk == P0 then route = [...]` 这种逻辑的执行，永远不如一行 Python 代码可靠。

**应该把所有逻辑都下沉到代码吗？** 不。有些"逻辑"是模糊的——比如"判断这个需求是否跨域"涉及对需求文本的理解，这是 LLM 的领地。边界在于：**输入是结构化的、规则是枚举的、输出是确定的→用代码；输入是自然语言的、判断需要理解力的、合理输出有多个→用 prompt。**

**框架约束 vs 个体技能。** 当前 AI Agent 领域的大多数讨论集中在"如何写更好的 prompt"——这相当于在没有编译器的年代讨论"如何写更少 bug 的汇编"。真正的杠杆在框架层面：类型系统、模块化、契约验证、确定性编排。个体的 prompt 技巧很快会被更好的框架淘汰，就像个体的汇编技巧被高级语言淘汰一样。ECW 做的事情——尝试构建一个 prompt 之上的治理框架——方向是对的。只是框架自身的实现也需要从 prompt 层上升到代码层。
