# 变更风险分级

> 本文档定义代码变更的风险分级体系。用于 `risk-classifier` skill 的自动判定，也供人工审阅时参考。

---

## 风险等级定义

| 等级 | 含义 | 流程要求 |
|------|------|---------|
| **P0 -- 极高** | 影响多域核心流程或外部系统契约 | 完整流程 + spec-challenge + 跨域 owner 确认 |
| **P1 -- 高** | 跨域影响或关键业务逻辑变更 | 完整需求澄清 + 标准 plan + biz-impact |
| **P2 -- 中** | 域内变更，有一定业务影响 | 简化需求确认 + plan + biz-impact |
| **P3 -- 低** | 无跨域影响，低业务风险 | 直接实现 + 测试验证 |

---

## 三维风险因子

```
总风险 = max(影响范围, 变更类型, 业务敏感度)
```

### 因子 1：影响范围

从依赖图自动推算。

| 级别 | 条件 | 项目实例 |
|------|------|---------|
| **P0** | 改了 6+ 域依赖的共享资源 | {your_shared_service_6plus}（如：UserCoreService 被 8 域依赖） |
| **P1** | 改了 4-5 域依赖的共享资源，或改了 MQ 消息体/Topic | {your_shared_service_4to5}（如：MessageBroker 被 5 域依赖） |
| **P2** | 改了 2-3 域依赖的共享资源，或跨域调用方法签名变更 | {your_shared_service_2to3}（如：CacheManager 被 2 域依赖） |
| **P3** | 域内变更，无跨域依赖 | 域内 Service 的私有方法、域内新增方法 |

**数据源：** `.claude/knowledge/common/shared-resources.md`

### 因子 2：变更类型

从 diff 模式识别。

| 级别 | 变更模式 | 检测方式 |
|------|---------|---------|
| **P0** | 状态机流转逻辑变更 | diff 涉及 `status`/`Status` 枚举值增减、状态更新方法逻辑、状态判断条件变更 |
| **P0** | 删除/重命名公开方法 | diff 中移除公开方法且该方法在跨域调用矩阵中被引用 |
| **P1** | 方法签名变更（参数增减、返回类型变化） | diff 中方法签名变化 + 该方法在其他域有调用 |
| **P1** | 数据写操作变更 | diff 中 SQL 写操作（UPDATE/DELETE/INSERT）或 ORM 写入逻辑变更 |
| **P2** | 新增公开方法 | diff 中新增公开方法，不影响已有调用方 |
| **P2** | 查询条件变更 | diff 中查询的 WHERE/过滤条件变更 |
| **P3** | 日志/注释/常量文案变更 | diff 仅涉及日志、注释、提示文案字符串 |
| **P3** | 新增私有方法 | diff 中新增私有方法 |

### 因子 3：业务敏感度

从域 + 操作类型推算。

| 级别 | 业务区域 | 原因 |
|------|---------|------|
| **P0** | {your_critical_resource_ops} | {reason}（如：库存增减 -- 错误 = 资损，多域依赖） |
| **P0** | 外部系统出站消息格式变更 | 对方已消费的消息格式变化 = 线上事故 |
| **P1** | {your_high_sensitivity_ops} | {reason}（如：任务状态流转 -- 多域依赖，卡死 = 全线阻塞） |
| **P1** | {your_core_entity_fields} | {reason}（如：订单核心字段 -- 影响财务对账和上下游同步） |
| **P2** | 配置驱动的逻辑分支 | 运行时行为可能和代码分析不一致 |
| **P2** | 异常处理逻辑 | 异常处理错误不阻塞主流程但影响人工介入效率 |
| **P3** | 查询/展示类接口 | 查询错误不影响数据，可快速修复 |

## 风险等级 -> 流程要求

| 流程步骤 | P0 极高 | P1 高 | P2 中 | P3 低 |
|---------|---------|-------|-------|-------|
| **需求澄清** | requirements-elicitation 完整流程 | requirements-elicitation 完整流程 | 简化确认（1 轮） | 跳过 |
| **方案设计** | writing-plans | writing-plans | writing-plans | 跳过 |
| **方案审查** | spec-challenge 对抗审查 | 人工审阅 plan | 人工审阅 plan | 跳过 |
| **跨域确认** | **必须**：所有受影响域 owner 签字 | 建议：通知受影响域 owner | 不需要 | 不需要 |
| **实现** | 逐步实现 + 每步 code-review | 标准实现 | 标准实现 | 直接实现 |
| **代码审查** | biz-impact + 人工 CR checklist 全项 | biz-impact + 人工 CR | biz-impact | 自动 CR |
| **测试验证** | 测试 + 端到端链路回归 + 灰度方案 | 测试 + 受影响域回归 | 测试 | 测试 |
| **上下游确认** | **必须**：外部系统团队确认 | 建议确认 | 不需要 | 不需要 |

## 人工覆盖

AI 自动分级后，人可以手动调整等级。以下场景需要人主动升级：

| 场景 | 动作 |
|------|------|
| 大促/高峰期冻结期内的任何变更 | 至少升至 P1 |
| 涉及运营口头约定但未文档化的规则 | 至少升至 P1 |
| 外部系统团队有发版窗口约束 | 至少升至 P1 |
| 人判断影响大于 AI 评估 | 人的判断优先 |

**升级是强制的，降级是建议的** -- 宁可多做不能少做。

## 紧急通道（Fast Track）

线上故障紧急修复时，不论风险等级，走简化路径：

| 流程步骤 | 紧急通道 |
|---------|---------|
| **风险预判** | Phase 1（保留，记录等级） |
| **需求确认** | 1 轮简化确认（3 个问题） |
| **方案设计** | writing-plans（精简版） |
| **方案审查** | 跳过 |
| **实现** | 直接实现 + 测试 |
| **影响分析** | biz-impact（实现后补做，标注 `[紧急通道]`） |

触发条件：用户说 "紧急修复"/"hotfix"/"线上问题"，或手动执行 `/risk-classify --hotfix`。
